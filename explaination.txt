**Overview**

This project is a Flask-based AI app generator that turns a natural-language prompt into a runnable starter application. It began as a simple form/code generator, but now it supports both:
- standard form-style modules like `login`, `registration`, `contact`, `profile`, `feedback`, `dashboard`
- richer CRUD-style business apps like `product_catalog`, `inventory_management`, `employee_management`, `customer_management`, `ticket_system`, `task_manager`, and generic `crud`

At a high level, the project does this:

1. The user opens the workspace UI and enters a prompt plus options.
2. The backend parses the prompt and decides what kind of module/app to generate.
3. It infers fields, intent, roles, workflows, and constraints.
4. It renders an HTML preview from templates.
5. It retrieves relevant snippets from the local snippet library.
6. It assembles backend code for the requested module/app type.
7. It validates the generated output.
8. It exports a runnable Flask project to `generated_projects/`.
9. It zips that project for download and shows the result in the UI.

**How The Request Flow Works**

**1. Frontend Workspace**
The app’s UI is a single workspace page. The user can:
- enter a free-text prompt
- choose a module type
- choose stack/framework
- provide project name and title
- define fields
- add notes/business rules
- enable or disable tests/readme/sample data

The frontend sends all of that to `/generate` in JSON.

That logic lives in:
- [workspace.html](D:\ai_code_generator\frontend\templates\workspace.html)
- [workspace-enhanced.js](D:\ai_code_generator\frontend\static\workspace-enhanced.js)

The workspace also shows:
- progress stages
- live preview
- generated HTML
- generated backend code
- validation report
- explanation of why the output was generated
- history of generated projects
- analytics on runs

---

**2. App Entry**
[app.py](D:\ai_code_generator\backend\app.py) is the backend entry point.

Its responsibilities:
- serve the workspace page
- expose sample prompts
- expose generation history
- expose a health endpoint
- accept `/generate`
- export ZIP downloads
- log generation success/failure

Important routes:
- `/`
- `/samples`
- `/history`
- `/health`
- `/generate`
- `/download`

So [app.py](D:\ai_code_generator\backend\app.py) is the orchestrator, but not where the real generation logic lives.

---

**3. Prompt Parsing**
[prompt_parser.py](D:\ai_code_generator\backend\prompt_parser.py) is where raw text becomes structured generation intent.

It detects:
- module type
- framework
- language
- fields
- roles
- constraints
- workflows
- entities
- intent
- resource name
- app family

Examples:
- “Build an employee management system...” becomes `employee_management`
- “Create a CRUD app for products...” becomes `crud` with resource `product`
- “Build a login form...” becomes `login`

It also groups some modules into an app family:
- `standard`
- `crud`

That distinction is important because CRUD-family modules share richer generation behavior.

CRUD-family modules currently include:
- `crud`
- `product_catalog`
- `inventory_management`
- `employee_management`
- `customer_management`
- `ticket_system`
- `task_manager`

Standard modules include:
- `login`
- `registration`
- `dashboard`
- `contact`
- `profile`
- `feedback`

---

**4. Field Inference**
[field_extractor.py](D:\ai_code_generator\backend\field_extractor.py) turns field names into richer field metadata.

For each field it decides:
- field name
- human label
- input type
- placeholder
- required vs optional
- help text

Examples:
- `email` -> `email`
- `password` -> `password`
- `phone` -> `tel`
- `description` -> `textarea`
- `quantity` -> `number`

This lets templates render sensible form controls without the user specifying HTML details.

---

**5. Template Selection + HTML Generation**
[template_selector.py](D:\ai_code_generator\backend\template_selector.py) chooses which template file to use.

Examples:
- `login` -> login template
- `registration` -> registration template
- `dashboard` -> dashboard template
- CRUD-family modules -> CRUD overview template
- generic modules -> generic form template

Then [code_generator.py](D:\ai_code_generator\backend\code_generator.py) renders the chosen Jinja template with:
- fields
- page title
- description
- project name
- resource names

This generates the preview HTML shown in the UI and also seeds the exported project.

---

**6. Snippet Retrieval**
This is the project’s RAG-like part.

Files:
- [rag_engine.py](D:\ai_code_generator\backend\rag_engine.py)
- [embedding_engine.py](D:\ai_code_generator\backend\embedding_engine.py)
- [vector_store.py](D:\ai_code_generator\backend\vector_store.py)

How it works:
- snippet metadata is loaded from [snippet_metadata.json](D:\ai_code_generator\data\snippet_metadata.json)
- code snippets are loaded from the `snippets/` folder
- embeddings are used when available
- retrieval is ranked by:
  - module
  - framework
  - intent
  - entities
  - workflows
  - query overlap

There is also an offline-safe fallback in [embedding_engine.py](D:\ai_code_generator\backend\embedding_engine.py), so the app still works even if transformer model downloads are unavailable.

This is important because it makes generation robust in restricted environments.

---

**7. Code Assembly**
[code_assembler.py](D:\ai_code_generator\backend\code_assembler.py) turns templates + snippets + parsed context into actual backend route code.

This file contains the real generation branching:
- login backend assembly
- dashboard backend assembly
- form backend assembly
- CRUD backend assembly

The CRUD backend is the most advanced path. It generates:
- protected list page
- create page
- detail page
- edit page
- delete action
- dashboard page
- login-required checks
- admin-only permissions for mutating actions

It also generates the “why this was generated” explanation shown in the frontend.

That explanation summarizes:
- selected module
- inferred fields
- detected roles
- workflows
- constraints
- chosen snippets

---

**8. Validation**
[code_validator.py](D:\ai_code_generator\backend\code_validator.py) checks the generated output.

It validates:
- Python syntax
- HTML structure
- imports
- route conflicts
- dependency inference
- security findings
- simple lint-style warnings
- route count
- overall quality score

So the validation output isn’t just pass/fail anymore. It gives structured diagnostics, which the frontend shows in the Validation tab.

---

**9. Project Export**
[project_generator.py](D:\ai_code_generator\backend\project_generator.py) is where generated code becomes a real runnable app.

This file:
- creates the folder structure
- writes routes
- writes templates
- writes CSS
- writes config
- writes storage helpers
- writes README/setup files
- writes tests
- seeds default data format

For standard modules, it exports a simpler Flask app.

For CRUD-family apps, it exports a richer project with:
- `app.py`
- auth route module
- CRUD route module
- storage service
- list/detail/form templates
- dashboard template
- login template
- seeded demo data
- tests
- setup docs

This is the piece that transformed the project from “code generator” into “starter app generator”.

**10. Download + History**

**Current Limitations**

It is much stronger now, but still has limits.

Main limitations:
- Flask-first in practice
- JSON-backed persistence by default, not full DB-backed production apps
- auth is demo/starter auth, not production security architecture
- some standard modules are still simpler than CRUD-family apps
- dashboards are starter dashboards, not advanced analytics systems
- snippet base is still small
- prompt understanding is smarter, but still mainly heuristic/rule-driven
- generated apps are starter apps, not complete production SaaS systems

So it’s best understood as:
- a strong Flask app starter generator
- not yet a full enterprise app platform
